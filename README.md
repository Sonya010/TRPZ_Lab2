# Частина 1. Дослідження контейнеризації Python-застосунку

## Дослідження 1: Базова (неоптимальна) збірка Python-застосунку

**Опис експерименту:** Створено базовий `Dockerfile` для пакування Python-застосунку без оптимізації кешування шарів. Мета полягала у вимірюванні початкового розміру образу та часу його збірки.

**Конфігурація середовища:**
* **Апаратна платформа:** MacBook Pro (Apple M3 Pro)
* **Базовий образ:** `python:3.13-bookworm`

**Відтворення:**
* **Файл конфігурації:** [Dockerfile для неоптимальної збірки](https://github.com/Sonya010/TRPZ_Lab2/blob/b6c53f9ec62782ca71465c61b6ae6f067c0e80c2/python-app/Dockerfile)
* **Команда збірки:** `docker build -t python-research:v1 .`

**Отримані результати:**
* **Час збірки образу:** 8.3 с
* **Розмір образу (Disk Usage):** 1.54 GB (повний розмір на диску разом із базовим образом Debian)
* **Розмір контенту (Content Size):** 388 MB (розмір доданих залежностей та коду застосунку)

**Аналіз та виявлені проблеми:**
Даний підхід є неоптимальним. Оскільки інструкція `COPY . .` виконується до інструкції `RUN pip install --no-cache-dir -r requirements/backend.in`, будь-яка зміна у вихідному коді застосунку буде інвалідувати кеш Docker. Це призводитиме до того, що при кожній зміні коду всі залежності завантажуватимуться та встановлюватимуться наново, що значно збільшить час наступних збірок. Також загальний розмір образу (1.54 GB) є надто великим через використання великого базового образу `bookworm`.

**Відтворення:**

* **Розмір образу (Disk Usage):** 1.54 GB

Залежності завантажувались і встановлювались наново, що підтверджує неефективність такого підходу.

## Дослідження 3: Оптимізація Dockerfile (Layer Caching)

**Опис експерименту:** `Dockerfile` було переписано для ефективного використання кешування шарів. Інструкції розділено: спочатку копіюються файли залежностей і виконується `pip install`, а потім копіюється вихідний код . Мета — перевірити зменшення часу збірки при зміні коду.

**Відтворення:**
* **Файл конфігурації:** [Оптимізований Dockerfile (Layer Caching)](https://github.com/Sonya010/TRPZ_Lab2/blob/8633eea5c77b4ea425e2a38347c4c79f4466b9a3/python-app/Dockerfile)
* **Команда збірки:** `docker build -t python-research:v2-optimized .`

**Отримані результати:**
* **Час збірки образу (після зміни коду):** 0.8 с
* **Розмір образу (Disk Usage):** 1.54 GB
* **Розмір контенту (Content Size):** 388 MB 

**Аналіз:**
Оптимізація пройшла успішно. Оскільки шар із встановленням залежностей (`RUN pip install`) тепер знаходиться до копіювання всього коду (`COPY . .`), зміна коду застосунку більше не інвалідує кеш залежностей. Docker використав закешований шар `pip install` (позначка `CACHED`), що зменшило час збірки з 9.1 с до 0.8 с. Розмір образу не змінився, оскільки базовий образ залишився тим самим.

**Опис експерименту:** Базовий образ у `Dockerfile` було змінено з важкого `python:3.13-bookworm` (на базі Debian) на легкий `python:3.13-alpine`. Мета — виміряти розмір фінального образу та час його збірки.

**Відтворення:**
* **Файл конфігурації:** [Dockerfile на базі Alpine](https://github.com/Sonya010/TRPZ_Lab2/blob/8d02f64d64b6896c6d306b019664ee9c43d78ac6/python-app/Dockerfile)
* **Команда збірки:** `docker build -t python-research:v3-alpine .`

**Отримані результати:**
* **Час збірки образу:** 11.5 с
* **Розмір образу (Disk Usage):** 137 MB
* **Розмір контенту (Content Size):** 32.6 MB 

**Аналіз:**
Використання `alpine` дозволило кардинально зменшити загальний розмір образу (з 1.54 GB до 137 MB). Це критично важливо для економії місця на серверах та пришвидшення деплою. Час збірки склав 11.5 с, оскільки змінився базовий образ і залежності встановлювалися з нуля.

**Опис експерименту:** До проєкту додано бібліотеку `numpy`, яка містить C-розширення. Було створено новий ендпоінт для перемноження матриць. Мета — порівняти час збірки та розмір образів на базі Alpine (`musl libc`) та Debian (`glibc`) при використанні "важких" залежностей.

**Відтворення:**

* **Команди збірки:**

**Отримані результати:**
* **Alpine + numpy:** час збірки 12.9 с, розмір образу (Disk Usage) 231 MB.

**Аналіз:**
Додавання `numpy` збільшило розмір обох образів, але різниця між ними залишається колосальною (231 MB проти 1.63 GB). Час збірки на даній апаратній платформі виявився майже однаковим, проте фундаментальна різниця в базових C-бібліотеках зберігається. Debian використовує стандартну бібліотеку `glibc` (standard `manylinux` wheels), під яку оптимізовано абсолютну більшість Python-пакетів. Alpine використовує мінімалістичну `musl libc`, що у багатьох випадках вимагає довгої компіляції складних пакетів з вихідного коду (якщо немає готових `musllinux` wheels) і може призводити до багів або зниження продуктивності в обчисленнях.

# Частина 2. Musl (Alpine) vs glibc (Ubuntu/Debian) на прикладі DNS

## Дослідження 6:

**Опис експерименту:** Для демонстрації різниці в роботі системних бібліотек (`glibc` в Ubuntu та `musl` в Alpine) було створено ізольовану Docker-мережу та піднято локальний DNS-сервер (dnsmasq). Сервер налаштовано на резолвінг домену `myservice.internal.corp` на IP-адресу `10.0.0.50`. З контейнерів Ubuntu та Alpine виконувався пошук неповного домену `myservice.internal` із використанням параметра `--dns-search="corp"`.

**Отримані результати:**
* **Ubuntu (glibc):** Успішно повернув IP-адресу `10.0.0.50` для `myservice.internal.corp`.
* **Alpine (musl libc):** Не зміг розрішити домен (пуста відповідь).

**Аналіз логів DNS-сервера:**
* Під час запиту від Ubuntu сервер отримав: `query[A] myservice.internal.corp`.
* Під час запиту від Alpine сервер отримав: `query[A] myservice.internal`.

**Пояснення результатів та наслідки:**
Поведінка відрізняється через реалізацію системного резолвера. Бібліотека `glibc` (Ubuntu/Debian) коректно обробляє директиву `search` (суфікс домену). Вона бачить, що пошук за абсолютним ім'ям не вдався, і автоматично підставляє суфікс `.corp`. 
Натомість `musl libc` (Alpine) по-іншому обробляє імена, що вже містять крапку (як `myservice.internal`). Вона сприймає їх як повністю визначені доменні імена (FQDN) і не намагається підставити `search`-домен.
У складних мікросервісних архітектурах сервіси часто покладаються на неповні DNS-імена та `search` суфікси. Переведення застосунку з `debian` на `alpine` заради зменшення розміру може призвести до того, що мікросервіси раптово втратять зв'язок один з одним (помилка "Name or service not known"), що дуже важко дебажити.

Наявність цих зайвих файлів не лише невиправдано збільшує вагу образу (що сповільнює деплой), але й створює додаткові ризики безпеки.

## Дослідження 8: Багатоетапна збірка (Multi-stage) з використанням FROM scratch

**Опис експерименту:** Для оптимізації застосовано підхід багатоетапної збірки (multi-stage build). На першому етапі (`builder`) у повноцінному образі `golang:1.22` компілюється статично слінкований двійковий файл (з прапорцем `CGO_ENABLED=0`). На другому етапі цей файл копіюється в абсолютно порожній базовий образ `scratch`.

**Відтворення:**
* **Файл конфігурації:** [Dockerfile для Multi-stage (scratch)](https://github.com/Sonya010/TRPZ_Lab2/blob/dc3a73ecb74e2b6d80e52dcc567a061e12645014/golang-app/Dockerfile)
* **Команда збірки:** `docker build -t golang-research:v2-scratch .`

**Отримані результати:**
* **Час збірки образу:** 9.0 с
* **Розмір образу (Disk Usage):** 16 MB
* **Розмір контенту (Content Size):** 5.6 MB

**Аналіз та відповіді на питання:**
* **Чи достатньо файлів для запуску?** Так, статичного бінарника достатньо, оскільки він не вимагає зовнішніх C-бібліотек.
* **Чи зручно виконувати дії всередині контейнера?** Ні, абсолютно незручно. Образ `scratch` не містить командної оболонки (`sh`/`bash`) та базових утиліт (`ls`, `cat`). Неможливо виконати `docker exec` для доступу в контейнер з метою дебагу. Також відсутні кореневі сертифікати (ca-certificates), що унеможливлює здійснення HTTPS-запитів на зовнішні ресурси. Цей підхід створює "чорний ящик".

## Дослідження 9: Багатоетапна збірка з використанням Distroless

**Опис експерименту:** Багатоетапну збірку змінено: замість повністю порожнього `scratch` на фінальному етапі використано образ `gcr.io/distroless/static-debian12` від Google. Мета — порівняти розміри та оцінити доцільність використання цього підходу для вирішення проблем порожнього контейнера.

**Відтворення:**
* **Файл конфігурації:** [Фінальний Dockerfile (Distroless з шаблонами)](https://github.com/Sonya010/TRPZ_Lab2/blob/07ca8b9362645ecadfa05a6787a281a87b3a4b07/golang-app/Dockerfile)
* **Команда збірки:** `docker build -t golang-research:v3-distroless .`

**Отримані результати:**
* **Час збірки образу:** 8.4 с
* **Розмір образу (Disk Usage):** 22.1 MB
* **Розмір контенту (Content Size):** 6.31 MB

**Аналіз та висновки:**
Розмір образу дещо збільшився у порівнянні зі `scratch` (з 16 MB до 22.1 MB). Ця різниця зумовлена тим, що образ `distroless` містить базові та необхідні для роботи застосунку системні компоненти: кореневі SSL-сертифікати (ca-certificates), дані про часові пояси (tzdata) та інформацію про системних користувачів (наприклад, файли `/etc/passwd`). 
Завдяки цьому застосунок здатний коректно виконувати HTTPS-запити в інтернет та працювати з часом. Водночас `distroless` не містить пакетних менеджерів, оболонки `sh`/`bash` чи інших утиліт, зберігаючи мінімальний розмір та максимально зменшуючи поверхню для потенційних хакерських атак (attack surface). Це робить його золотим стандартом для пакування скомпільованих мов (Go, Rust).

# Загальні висновки та рекомендації щодо контейнеризації

На основі проведеної серії експериментів із Python та Golang застосунками, можна сформувати наступні найкращі практики для пакування застосунків у Docker-контейнери:

1. **Оптимізація кешування шарів (Layer Caching):** Порядок інструкцій у `Dockerfile` має критичне значення для швидкості CI/CD. Інструкції, які змінюються найрідше (наприклад, встановлення системних пакетів або Python-залежностей через `requirements.txt`), повинні знаходитися якомога вище. Копіювання вихідного коду (`COPY . .`), який змінюється постійно, має виконуватися останнім. Це дозволяє скоротити час збірки з десятків секунд до мілісекунд.
2. **Обережний вибір базового образу (Alpine vs Debian):**
   Використання мінімалістичного `alpine` не завжди є срібною кулею. 
   * Для інтерпретованих мов (Python), якщо проєкт використовує бібліотеки з C-розширеннями (як `numpy`, `pandas`), краще використовувати `slim` версії на базі Debian (`glibc`). Використання Alpine (`musl`) у таких випадках призводить до довгої компіляції та ризикує стабільністю, зводячи нанівець перевагу в розмірі.
   * Перехід на Alpine також може зламати внутрішню маршрутизацію в мікросервісах через специфіку роботи `musl libc` із DNS (ігнорування `search` доменів для FQDN-подібних запитів).
3. **Обов'язкове використання Multi-stage збірок для компільованих мов:**
   Для мов типу Go, Rust чи C++ базовий образ потрібен лише для компіляції. Тягнути компілятор та вихідний код у продакшен — це марнотратство ресурсів та загроза безпеці. Багатоетапна збірка дозволяє зменшити розмір образу у десятки разів (в моєму експерименті з 1.33 GB до 22 MB).
4. **Використання Distroless замість Scratch:**
   Хоча порожній образ `scratch` дає мінімальний розмір, його використання створює проблеми в продакшені (відсутність SSL-сертифікатів для HTTPS, відсутність таймзон, неможливість дебагу). Образи `distroless` від Google є золотою серединою: вони містять необхідний системний мінімум, але позбавлені shell-оболонок та пакетних менеджерів, що робить їх максимально безпечними та легкими.
