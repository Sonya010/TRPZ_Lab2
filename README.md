### Дослідження 1: Базова (неоптимальна) збірка Python-застосунку

**Опис експерименту:** Створено базовий `Dockerfile` для пакування Python-застосунку без оптимізації кешування шарів. Мета полягала у вимірюванні початкового розміру образу та часу його збірки.

**Конфігурація середовища:**
* **Апаратна платформа:** MacBook Pro (Apple M3 Pro)
* **Базовий образ:** `python:3.13-bookworm`

**Відтворення (команда збірки):**
`docker build -t python-research:v1 .`

**Отримані результати:**
* **Час збірки образу:** 8.3 с
* **Розмір образу (Disk Usage):** 1.54 GB (повний розмір на диску разом із базовим образом Debian)
* **Розмір контенту (Content Size):** 388 MB (розмір доданих залежностей та коду застосунку)

**Аналіз та виявлені проблеми:**
Даний підхід є неоптимальним. Оскільки інструкція `COPY . .` виконується до інструкції `RUN pip install --no-cache-dir -r requirements/backend.in`, будь-яка зміна у вихідному коді застосунку буде інвалідувати кеш Docker. Це призводитиме до того, що при кожній зміні коду всі залежності завантажуватимуться та встановлюватимуться наново, що значно збільшить час наступних збірок. Також загальний розмір образу (1.54 GB) є надто великим через використання великого базового образу `bookworm`.

### Дослідження 2: Вплив зміни коду на неоптимальну збірку

**Опис експерименту:** Внесено невелику зміну у вихідний код застосунку (додано коментар). Мета — виміряти новий розмір образу та час його збірки, щоб перевірити роботу кешування Docker.

**Відтворення:**
`docker build -t python-research:v1-changed .`

**Отримані результати:**
* **Час збірки образу:** 9.1 с
* **Розмір образу (Disk Usage):** 1.54 GB
* **Розмір контенту (Content Size):** 388 MB 

**Аналіз:**
Як видно з результатів, зміна лише одного рядка коду призвела до збільшення часу збірки (9.1 с). Оскільки інструкція `COPY . .` стоїть перед `RUN pip install`, Docker зафіксував зміну контексту і скинув кеш для всіх наступних шарів. Залежності завантажувались і встановлювались наново, що підтверджує неефективність такого підходу.

### Дослідження 3: Оптимізація Dockerfile (Layer Caching)

**Опис експерименту:** `Dockerfile` було переписано для ефективного використання кешування шарів. Інструкції розділено: спочатку копіюються файли залежностей і виконується `pip install`, а потім копіюється вихідний код . Мета — перевірити зменшення часу збірки при зміні коду.

**Відтворення:**
`docker build -t python-research:v2-optimized .`

**Отримані результати:**
* **Час збірки образу (після зміни коду):** 0.8 с
* **Розмір образу (Disk Usage):** 1.54 GB
* **Розмір контенту (Content Size):** 388 MB 

**Аналіз:**
Оптимізація пройшла успішно. Оскільки шар із встановленням залежностей (`RUN pip install`) тепер знаходиться до копіювання всього коду (`COPY . .`), зміна коду застосунку більше не інвалідує кеш залежностей. Docker використав закешований шар `pip install` (позначка `CACHED`), що зменшило час збірки з 9.1 с до 0.8 с. Розмір образу не змінився, оскільки базовий образ залишився тим самим.

### Дослідження 4: Зменшення розміру образу (Alpine Linux)

**Опис експерименту:** Базовий образ у `Dockerfile` було змінено з важкого `python:3.13-bookworm` (на базі Debian) на легкий `python:3.13-alpine`. Мета — виміряти розмір фінального образу та час його збірки.

**Відтворення:**
`docker build -t python-research:v3-alpine .`

**Отримані результати:**
* **Час збірки образу:** 11.5 с
* **Розмір образу (Disk Usage):** 137 MB
* **Розмір контенту (Content Size):** 32.6 MB 

**Аналіз:**
Використання `alpine` дозволило кардинально зменшити загальний розмір образу (з 1.54 GB до 137 MB). Це критично важливо для економії місця на серверах та пришвидшення деплою. Час збірки склав 11.5 с, оскільки змінився базовий образ і залежності встановлювалися з нуля.
